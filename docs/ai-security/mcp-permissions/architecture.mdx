# Agent Security Architecture

## What is Agent Security?

Agent Security delivers **permissions-as-a-service** for any Model Context Protocol (MCP) server. It acts as a proxy/gateway between AI agents and the MCP server resources they interact with, providing:

* **Fine-grained ReBAC** (relationship-based access control) powered by Permit.io, OPA, and OPAL
* **Authentication & authorization** that binds user, agent, MCP server, and downstream service identities
* **Comprehensive auditing** covering every tool call, user, and agent
* **Flexible deployment options** *(hosted gateway available; additional patterns coming soon)*, all using the same policy rules

A single URL switch yields consistent enforcement, instant visibility, and human-in-the-loop (HITL) approvals *(coming soon)* â€” **no code changes** to agents or servers.

**Key components:**
- **Platform** (`app.agent.security`): Admin interface for managing hosts, MCP servers, and user access
- **Edge Router** (NGINX): Routes requests by path to Gateway or Consent Service
- **Gateway** (`*.agent.security/mcp`): MCP proxy that enforces authorization on every tool call
- **Consent Service** (`*.agent.security`): Handles OAuth login, user consent, and upstream MCP OAuth
- **Permit.io PDP**: Policy decision point for real-time authorization checks
- **Redis**: Session and host configuration store (accessed only by Gateway)
- **PostgreSQL**: User accounts, OAuth clients, and auth sessions

:::info Architecture constraint
The Consent Service does **not** access Redis directly. All state persistence goes through the Gateway Admin API, centralizing Redis schema and logic in the Gateway.
:::

## Policy Model

### Trust-Level Access Control

Agent Security classifies each tool into a trust level based on its risk:

1. **Low trust** â€” everything else (read-only by convention); this is the default for tools not matching medium or high patterns
2. **Medium trust** â€” write operations (tools containing: `create`, `write`, `update`, `set`, `modify`, `edit`, `put`, `post`, `insert`, `add`, `send`, `execute`, `run`, `invoke`, `submit`, `push`, `publish`, `deploy`, `apply`, `patch`)
3. **High trust** â€” destructive operations (tools containing: `delete`, `remove`, `destroy`, `drop`, `purge`, `erase`, `truncate`, `terminate`, `kill`, `revoke`)

Trust levels are hierarchical: higher levels inherit all permissions from lower levels.

### Planned Policy Enhancements *(coming soon)*

- **Organization Policy** â€” control agent-type â†” server-type pairings
- **Operation Approval** â€” per-call HITL step-up
- **Org-Wide Constraints** â€” DLP masks, region pinning, rate limits

### Policy Architecture

Agent Security automatically generates [Google-Zanzibar](/modeling/google-drive)-inspired ReBAC (Relationship-based Access Control) policies based on:
- Defined roles for users and agents
- MCP server resource instances
- Agent roles derived from the user's consent choices
- A `user_profile` resource that links each user to the agents acting on their behalf, enabling relationship-based permission derivation through a 3-way chain: **Agent** --role on--> **UserProfile** --relation to--> **MCP Server**. The effective permission on a server is derived as `min(agent_role_on_profile, profile_relation_to_server)` â€” meaning a human's profile relation acts as a ceiling on the trust level any agent can actually exercise, regardless of what trust level was granted during consent.

Each MCP server maps to a **Permit resource type** whose key is the server key (e.g., `linear`). The server's tools become **actions** on that resource, using slugified tool names (e.g., `create_issue`). There are no separate resource instances per tool â€” the server itself is both the resource type and instance.

```mermaid

    S --Action--> F1[get_issues]
    S --Action--> F2[create_issue]
    S --Action--> F3[delete_repo]
```

## Authentication & Authorization

### How it Works

**Admin setup (one-time):**

1. Admin creates a host at [app.agent.security](https://app.agent.security), linked to a Permit environment
2. Admin imports upstream MCP servers, classifies tools by trust level, and sets max trust levels per user
3. Admin grants specific users permission to connect to specific MCP servers

**User connection (first time):**

1. The MCP client discovers the gateway's OAuth endpoints
2. The user authenticates and signs in
3. The user selects an MCP server from the list of servers the admin has granted them access to
4. If the upstream MCP server requires OAuth, the user authorizes with the upstream provider
5. The user chooses the trust level they want to grant their agent, up to the maximum the admin configured
6. The gateway issues a JWT access token to the MCP client

**On subsequent tool calls:**

1. The gateway verifies the JWT and identifies the agent
2. The gateway checks Permit: *"Can this agent call this tool on this MCP server?"*
3. Allowed calls are proxied; denied calls return a permission denied error

**Identity format:** The gateway constructs a user key for the Permit check based on the caller type:
- **Human users**: `human|{subject}` â€” where `subject` is the authenticated user's identity. This format is used by the consent service and platform for policy management (e.g., granting a user access to MCP servers).
- **Agents**: `agent|{client_id}` â€” where `client_id` identifies the agent acting on behalf of the user. This format is used at runtime by the gateway for tool-call authorization checks.

### Sequence: Admin Setup Flow

The admin creates a host through the Platform UI, which resolves the Permit API key, provisions the host configuration in the Gateway, and stores it in Redis.

When a user connects for the first time, the MCP client discovers OAuth endpoints, the user authenticates and consents, and if the upstream MCP server requires OAuth, that flow is handled transparently.

:::info Server allow-list
Users can only connect to MCP servers that the admin has pre-configured and granted them access to. Ad-hoc upstream URLs are not accepted â€” the consent service validates that every upstream URL matches a server the user is authorized to use.
:::

### Sequence: Authorization Decision Flow

When a tool call arrives at the Gateway, it passes through a multi-step middleware chain before reaching the upstream MCP server.

```mermaid
sequenceDiagram
    participant C as MCP Client
    participant GW as Gateway
    participant R as Redis
    participant PDP as Permit PDP
    participant Up as Upstream MCP Server

    C->>GW: call_tool(name=create_issue, Bearer token)

    Note over GW: 1. Extract tenant from Host header
    Note over GW: Host: acme.agent.security â†’ tenant "acme"

    GW->>R: GET host:acme
    Note over GW,R: 2. Load tenant config (permit_api_key, oauth config)

    Note over GW: 3. JWT Validation
    Note over GW: Verify Ed25519 signature (JWKS cached)
    Note over GW: Validate claims: iss, aud, exp, tenant_subdomain
    Note over GW: Extract: subject (user ID), client_id (agent ID)

    Note over GW: 4. Transport session lookup (in-memory)
    Note over GW: MCP-Session-Id â†’ local HashMap

    GW->>R: GET mcp:session:{client_id}:{subject}
    R-->>GW: {upstream_url, tokens, trust_level}
    Note over GW,R: 5. Application session lookup

    Note over GW: 6. Token refresh (if upstream token expires within 5 min)

    GW->>PDP: permit.check("agent|{client_id}", "create_issue", "{server_key}:{server_key}")
    Note over GW,PDP: 7. Authorization check

    alt allowed
        PDP-->>GW: ALLOW
        GW->>Up: call_tool (proxied with upstream token)
        Up-->>GW: tool result
        GW-->>C: tool result
    else denied
        PDP-->>GW: DENY
        GW-->>C: MCP error (-32001 Permission denied)
    end
    GW->>GW: Log to audit
```

### Authorization: Trust Ceiling (min logic)

The effective permission on an MCP server is determined by the **minimum** of the agent's trust level and the human's profile relation to the server. This ensures a human's profile relation acts as a ceiling on what any agent can exercise.

```mermaid
flowchart LR
    Agent["Agent<br/>agent|client_id"]
    Profile["UserProfile<br/>user_profile:subject"]
    Server["MCP Server<br/>server_key:server_key"]
    Result["Effective Role<br/>= min(agent_role, profile_relation)"]

    Agent -->|"role: {server}-medium"| Profile
    Profile -->|"relation: medium"| Server
    Server --> Result

    style Result fill:#f9f,stroke:#333,stroke-width:2px
```

**How it works:**

The system uses 9 derived role rules per MCP server to implement the `min()` logic through Permit's ReBAC:

| Agent role on profile | Profile relation to server | Effective server role |
| --- | --- | --- |
| `{server}-high` | `high` | **high** |
| `{server}-high` | `medium` | **medium** (capped) |
| `{server}-high` | `low` | **low** (capped) |
| `{server}-medium` | `high` | **medium** |
| `{server}-medium` | `medium` | **medium** |
| `{server}-medium` | `low` | **low** (capped) |
| `{server}-low` | `high` | **low** |
| `{server}-low` | `medium` | **low** |
| `{server}-low` | `low` | **low** |

The derived role then determines which tools are allowed:
- **Low** trust tools: available to `low`, `medium`, and `high` roles
- **Medium** trust tools: available to `medium` and `high` roles
- **High** trust tools: available to `high` role only

## Key Advantages

* **Single control point** for authorization, audit, and anomaly detection (planned)
* **Drop-in** â€” no code changes, SDK optional
* **Fine-grained ReBAC** â€” captures true user-agent-resource relationships
* **Policy-as-code** â€” Rego, GitOps, unit tests
* **Short-lived creds** â€” minimize blast radius
* **HITL safety net** *(coming soon)* â€” step-up approvals for risky ops
* **DLP hooks** *(coming soon)* â€” mask/redact before LLM exposure
