# Human-in-the-Loop Approvals

Human-in-the-Loop (HITL) adds a **real-time approval gate** to your agent workflows. When an agent requests a sensitive operation — such as deleting records, modifying production data, or accessing restricted resources — the gateway pauses execution and routes an approval request to a designated admin. Routine operations continue without interruption.

Permit MCP Gateway sits between your MCP clients and upstream servers. HITL extends this by pausing a tool call mid-flight, notifying an approver, and resuming or rejecting based on their decision — all transparent to the MCP client.

:::note Enterprise Plan
Human-in-the-Loop approvals are available on **Enterprise** plans. [Schedule a demo](https://calendly.com/permit-io/demo) to get started.
:::

:::tip Quick start
1. Go to **MCP Servers** → select a server → toggle **HITL** on a tool
2. Trigger that tool from an MCP client (e.g., Cursor, Claude Desktop)
3. Open **Approvals** in the Platform → approve or reject the request
:::

## How it works

1. An agent calls a tool that requires approval (e.g., `delete_repo`)
2. The gateway checks authorization via Permit.io (existing [trust-level](/permit-mcp-gateway/guide) check)
3. If the tool is approval-gated, the gateway **pauses the request** and creates an approval request
4. Admins are notified via the Platform UI, email, and/or Slack
5. An admin reviews the request and **approves or rejects** it
6. The gateway resumes the tool call (on approval) or returns an error (on rejection/timeout)

The MCP client simply waits for the response — no client changes are needed. If no decision is made within the timeout (default: 5 minutes), the request is automatically rejected (fail-closed).

## Approval queue

The **Approvals** page in the Platform is where admins review and act on pending requests. It has three tabs: **Pending**, **History**, and **Notifications**.

Any user with access to the Platform can review and approve or reject pending requests.

### Pending approvals

Each pending approval card shows the full context needed to make a decision:

- **Tool name** and **server** — what the agent wants to do and where
- **Agent and user identity** — who is making the request
- **Tool arguments** — the exact parameters being passed (e.g., which issue ID, which repository). Very large arguments (over 64 KB) are automatically truncated for display.
- **Countdown timer** — time remaining before automatic rejection, with color indicators (green → amber → red as time runs out)

![Approval queue with pending request](/img/agent-security/hitl/approval-queue-pending.png)

Admins can **Approve** or **Reject** each request. When rejecting, you can provide a reason — either a quick preset ("Unauthorized scope", "Suspicious arguments", "Wrong environment") or a custom message.

![Rejecting with a reason](/img/agent-security/hitl/approval-reject-with-reason.png)

The rejection reason is included in the error message returned to the agent, so the user understands why their request was denied.

![Agent sees rejection reason](/img/agent-security/hitl/agent-rejection-message.png)

:::tip Keyboard shortcuts
Power users can navigate the approval queue with keyboard shortcuts: **J/K** to navigate between cards, **A** to approve, **R** to reject, **X** to select for batch actions, **Esc** to deselect.
:::

### Batch actions

When multiple approvals are pending (common in multi-tool agentic workflows), select multiple cards using the checkboxes and use the batch action bar to approve or reject all selected requests at once.

### Approval history

The **History** tab shows all resolved approvals with the following columns:

- **Time** — when the decision was made
- **Tool** and **Server** — what was requested
- **Agent** — which agent made the request
- **Decided By** — the admin who approved or rejected (blank for timeouts and cancellations)
- **Decision Time** — how long the approval waited before a decision was made
- **Decision** — the outcome: Approved, Rejected, Timed out, or Cancelled

![Approval history](/img/agent-security/hitl/approval-history-full.png)

Use the status filter to narrow down by outcome (e.g., show only rejected requests for review).

## Configuring approval policies

HITL approval can be configured at three levels, from most specific to broadest:

### Per-tool approval

On any MCP server's detail page, each tool has a **HITL toggle**. Enable it to require approval for that specific tool. This is the most granular control — use it for individual high-risk operations like `delete_repo` or `deploy_production`.

![Per-tool HITL toggle on the tool list](/img/agent-security/hitl/per-tool-hitl-toggle.png)

In the example above, `get_issue` has the HITL toggle enabled (amber) — any agent calling this tool will require admin approval. The other tools have HITL disabled and will execute normally based on their trust level.

### Server approval policy

On the MCP server detail page, the **Server Approval Policy** card provides broader controls:

- **Require approval for all tools** — when enabled, every tool on this server requires approval regardless of individual tool settings
- **Trust level threshold** — set a minimum [trust level](/permit-mcp-gateway/guide) that triggers approval. For example, set the threshold to "High" to require approval only for high-trust tools (least restrictive), or set it to "Low" to require approval for all tools (most restrictive)

![Server approval policy and trusted agent bypass](/img/agent-security/hitl/server-policy-and-bypass.png)

### How policies are evaluated

When an agent calls a tool, the gateway evaluates approval policies in this order:

1. **Trusted agent bypass** — if the agent is in the bypass list, the tool executes immediately (no approval needed)
2. **Per-tool toggle** — if the specific tool has HITL enabled, approval is required
3. **Server "require all" policy** — if enabled, all tools on this server require approval
4. **Trust level threshold** — if the tool's trust level meets or exceeds the threshold, approval is required

If **any** policy (2, 3, or 4) requires approval and the agent is **not** bypassed, the tool call is paused for approval.

### Trusted agent bypass

For trusted automation (e.g., CI bots, scheduled pipelines) that should not require manual approval, add the agent's client ID to the **Trusted Agent Bypass** list on the server detail page. Bypassed agents skip the approval check entirely for all tools on that server — this takes priority over all other policies.

You can find an agent's client ID on the **Agents** page in the Platform.

## Notifications

Configure where approval notifications are sent so admins know immediately when a request is waiting. Go to **Approvals > Notifications** to set up:

![Notification settings](/img/agent-security/hitl/notification-settings.png)

### Email notifications

Add email addresses to receive an approval notification email whenever a new request is created. The email includes the tool name, server, agent identity, and a link to review in the Platform.

![Email notification](/img/agent-security/hitl/email-notification.png)

### Slack notifications

Paste a [Slack Incoming Webhook](https://api.slack.com/messaging/webhooks) URL to receive notifications in a Slack channel. The message includes the tool name, server, agent, and user — plus a button to open the Platform for review.

For security, **tool arguments are not included** in Slack messages. To view the full arguments, click through to the Platform.

![Slack notification](/img/agent-security/hitl/slack-notification.png)

### Browser notifications

When the Platform is open in a browser tab (even in the background), you'll receive desktop notifications for new approval requests automatically. No configuration needed — the browser will ask for notification permission on your first visit to the Approvals page.

## Timeouts and fail-closed behavior

- **Default timeout:** 5 minutes — if no admin responds, the request is automatically rejected
- **Extend:** When reviewing a complex request, click the **Extend** button on the approval card to add more time. Each extension can add up to 30 minutes, with a maximum total lifetime of 1 hour
- **Pending limit:** A maximum of 50 approval requests can be pending per host at any time. If the limit is reached, new approval-gated tool calls are rejected until existing requests are resolved
- **Fail-closed:** Timeouts always result in rejection, never automatic approval
- **Client disconnect:** If the agent disconnects while waiting, the request is automatically cancelled to avoid wasting approver time

:::info Notification scope
Notification settings (email and Slack) apply to all MCP servers within the current host. Approval policies (per-tool, per-server, trusted agent bypass) are configured per server.
:::

## What the agent user sees

When a tool call is paused for approval, the agent receives a notification:

> Tool 'delete_repo' requires admin approval before execution. Waiting for approval (timeout: 5 minutes)...

Most MCP clients (Cursor, Claude Desktop, Claude Code) display this message to the user, so they know the wait is intentional — not a hang or bug. If the request is rejected, the error message includes the reason provided by the admin.
